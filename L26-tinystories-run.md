# L23 — Capstone: Train a GPT on TinyStories

This is the capstone. Everything you built across the course now snaps together
into one pipeline that trains a real — if tiny — language model on real text and
samples stories out of it. The pieces already exist; nothing new gets invented
here. A **language model** is a model that, given the text so far, predicts the
next token; train it on enough text and it learns to continue a prompt into
fluent English. Across M1–M4 you built every part of one: a numerically-stable
**softmax** (the function that turns a row of raw scores into a probability
distribution that sums to 1), **byte-level BPE** (the tokenizer that chops text
into integer ids), a **causal attention head** (the layer that lets each position
look only at earlier positions), the full **GPT block**, and a **training loop**
that takes optimizer steps and checkpoints its progress to disk. This lesson is
the **orchestration**: wiring those finished parts into one end-to-end run.

So this lesson does not re-teach the components — it references them and connects
them. Your job is the glue: re-derive the tokenizer cold (a memory check), then
write the one new piece the run needs — a `DataLoader` that turns the tokenized
corpus into training batches. Everything else (the GPT, the loop, the sampler) is
imported, frozen, and already tested.

## What "TinyStories" is and why we use it

**TinyStories** is a dataset of short synthetic children's stories — text like
*"Once upon a time there was a little dog. The dog liked to run."* It was built so
that the vocabulary and grammar stay simple: small word list, short sentences,
basic plots. That simplicity is the whole reason we train on it. A full-size
language model needs billions of parameters and weeks of compute to write fluent
English about *anything*; but the slice of English in TinyStories is narrow
enough that a model of only a few **million** parameters — small enough to train
overnight on a laptop — can learn to produce coherent, grammatical little stories.
It is the smallest real target that still gives a satisfying payoff: text that
reads like English, generated by a model you built and trained yourself.

Be honest about the scale, though. The model here is tiny by modern standards
(roughly 2.7 million parameters; today's frontier models are a million times
larger). Expect it to babble *plausible* children's-story English — grammatical
sentences, simple narrative shape — not deep plots or factual accuracy. The win is
coherence at this size, not brilliance.

## The end-to-end pipeline

The capstone run is four stages, each leaning on parts from earlier lessons. Read
this as the map; the rest of the lesson fills in the one stage you implement.

1. **Tokenize the data.** Train a byte-level BPE tokenizer on the raw TinyStories
   text, then use it to convert every story into a stream of integer token ids,
   saved to disk. Raw text is characters; the model only ever sees integers, so
   this is the bridge between them. (Tokenizer from L15; the warm-up below.)
2. **Build the GPT.** Construct `references.gpt.GPT` from a config — the
   decoder-only transformer you assembled in L20. Nothing to write; you import it.
3. **Train.** Feed the token stream to the model in batches and run
   `references.train_loop.train_loop` (L16 onward): each step predicts the next
   token, measures the loss, and nudges the parameters downhill, checkpointing
   periodically so a multi-hour run can be interrupted and resumed. The one new
   part you write — the `DataLoader` — sits between the token stream and this loop.
4. **Sample.** Once trained, generate fresh stories by repeatedly asking the model
   for a next-token distribution and drawing from it, using the `sample` function
   you wrote in L22. This is the payoff: watch your model tell a story.

Stages 2–4 are imported, finished code. **What you implement in this lesson is two
things:** the byte-level BPE tokenizer (stage 1), re-derived from memory as a
warm-up, and the `DataLoader` (the seam in stage 3) that slices the token stream
into next-token training batches. The rest of the lesson is those two pieces.

## The warm-up: byte-level BPE, recalled cold

Before new material, recall old material cold — re-deriving a thing from memory
fixes it far better than re-reading it. You last wrote **byte-level BPE** in L15.
BPE ("byte-pair encoding") is the tokenizer that learns to merge frequently-seen
adjacent byte pairs into single new tokens, so common chunks of text become one id
instead of many. Re-implement it now from memory — `train_bpe`, `encode`,
`decode` — without looking back. Two disciplines the warm-up pins, because both
are easy to get subtly wrong:

- **Train on the raw UTF-8 BYTES, not str characters.** Start from
  `list(text.encode("utf-8"))` (the text's bytes), never `list(text)` (its
  characters). The **base vocabulary** is the 256 possible byte values, ids
  `0..255`; every merge the trainer learns mints a new id `>= 256`. A tokenizer
  that builds its base vocab from str characters silently breaks on any multibyte
  input — accents, emoji, anything non-ASCII — because one such character is
  several bytes, and a character-based vocab has no way to represent it.
- **Encoding applies the learned RULES in RANK order.** Encoding is not re-running
  training. After training, you have an ordered list of merge rules — rank 0 is the
  first merge learned, rank 1 the next, and so on. To encode a new string, among
  the byte pairs *present in it right now*, always apply the rule learned
  **earliest** (lowest rank / lowest merge id), not the pair that happens to be
  most frequent in this particular string. Picking by encode-time frequency gives
  a different, wrong tokenization, because it ignores the order the rules were
  learned in.

This warm-up is graded first, on toy data, in under a second: exact round-trips
(encode then decode returns the original text), the exact merge sequence a fixed
corpus produces, and real compression (the encoded id stream is much shorter than
the raw byte stream).

## The DataLoader: turning tokens into training batches

Stage 3 needs a steady supply of training examples, and that is the one new piece
you write. After tokenizing, the whole corpus is a single long 1-D array of
integer ids sitting on disk. The training loop cannot consume that directly — it
needs **batches**: small rectangular blocks of `(batch_size, block_size)` tokens,
where `batch_size` is how many independent sequences to process at once and
`block_size` is how many tokens long each sequence is (the model's context
window). The `DataLoader` is the adapter that cuts those blocks out of the long
token array.

The task it solves is **next-token prediction**. At every position the model is
asked: given the tokens up to here, what comes next? So each training example is a
pair `(x, y)` where `y` is `x` shifted forward by exactly one position — `y[t]` is
the token that actually followed `x[t]`. That one-position shift is the entire
supervision signal: the answer to "what's next?" is just the next token, already
sitting in the corpus.

Here is `DataLoader.get_batch()` as an explicit procedure. To build one batch:

1. Compute `high = len(tokens) - block_size - 1`. A window needs `block_size + 1`
   tokens (the `block_size` inputs plus the one-position-shifted target, which
   reaches index `i + block_size`), so the last start that stays in range is
   `len(tokens) - block_size - 1` — exactly `high`.
2. Draw `batch_size` random start positions `i` with **`torch.randint(0, high,
   (batch_size,))` over the global torch RNG** (more on why this exact RNG below).
   `torch.randint`'s upper bound is exclusive, so this draws `0 <= i < high`. (That
   leaves the single last safe start, `i = high`, unsampled — a harmless one-window
   margin on a corpus of millions of tokens, and the convention the grader checks
   against.)
3. For each start `i`, slice the inputs `x = tokens[i : i + block_size]` and the
   targets `y = tokens[i + 1 : i + 1 + block_size]` — `y` is `x` shifted one
   position forward.
4. Stack the per-window slices into two `(batch_size, block_size)` `LongTensor`s
   (PyTorch tensors of 64-bit integers, the dtype the model's embedding lookup
   expects) and return `(x, y)`.

A worked check on a tiny example. Suppose `tokens = [0, 1, 2, 3, 4, 5, 6, 7]`
(eight consecutive ids), `block_size = 3`, and one window starts at `i = 2`. Then
`x = tokens[2:5] = [2, 3, 4]` and `y = tokens[3:6] = [3, 4, 5]`. Read the pairs
down the columns: input `2` → target `3`, input `3` → target `4`, input `4` →
target `5`. Every target is exactly its input plus one position later in the
stream — which on a consecutive-id corpus like this means `y == x + 1` element by
element. That is precisely the invariant the grader checks.

### The RNG discipline that makes exact-resume work

A multi-hour run must survive being interrupted: you checkpoint partway, kill it,
relaunch, and the resumed run must continue *exactly* as if it had never stopped —
same loss numbers, same batches. The frozen `references.train_loop` provides that
guarantee, but only if the `DataLoader` cooperates on one specific point: **where
its random window starts come from.**

A **random number generator (RNG)** is the source of the "random" start positions
in step 2. PyTorch has one shared **global** RNG, and `train_loop` seeds it at
entry and saves its internal state into every checkpoint. So if the `DataLoader`
draws from that global RNG — `torch.randint(0, high, (batch_size,))` with **no**
`generator=` argument — then the exact sequence of batches is part of what the
checkpoint captures, and a resumed run replays the identical batch stream.

The trap is to instead create a **private** generator — `torch.Generator()` — and
draw from it. Shapes and the `y = x + 1` shift still come out right, so the batch
test passes; but `train_loop` never sees that private generator, so its state is
*not* in the checkpoint. After a reload the private generator picks up wherever it
left off in memory (or restarts), the windows differ, and the resumed loss curve
diverges from the uninterrupted one. The fix is simply to use the global RNG. (The
grader hunts this exact mistake by name — see the named trap below.)

## Tokenizing the real corpus: the BPE-4k prep

The warm-up is the *idea* of the tokenizer; the real run needs a *production*
version of stage 1, and that is a one-time prep step you run from the command
line, not graded code you write. The non-graded script
`prep/train_bpe_4k.py` trains a **4096-token** byte-level BPE on the full
TinyStories text — an order of magnitude larger than the toy vocab of the warm-up —
using the canonical `references.bpe` (the same byte-level BPE, kept as one source
of truth, so the prep does not re-implement what you just wrote). A larger vocab
learns longer merges (common words and word-pieces become single tokens), so each
story compresses to far fewer tokens, which means the GPT sees more text per
training step. The script does four things:

1. Trains a 4k-vocab BPE on `datasets/tinystories/data/train.txt`.
2. Writes the tokenizer to `bpe4k_merges.pkl` + `bpe4k_vocab.json`.
3. Tokenizes train and val into `train_tokens.bin` / `val_tokens.bin` — `uint16`
   numpy memmaps (explained next).
4. Logs the token count and the compression ratio versus char-level.

Those token `.bin` files are exactly what your `DataLoader` will open and slice
into batches for the overnight run.

### Why uint16 and why a memmap

The tokenized corpus is millions of integers, and two storage choices keep it
cheap:

- **`uint16`, not `int32`.** `uint16` is an unsigned 16-bit integer — it can hold
  values `0..65535`. The largest token id is `vocab_size - 1 = 4095`, comfortably
  under that ceiling, so 16 bits per token is plenty. `int32` would use 32 bits for
  the same numbers, doubling the file size and the memory traffic for no benefit.
- **A memmap, not an in-RAM array.** A **memmap** (`np.memmap`, "memory-mapped
  file") is an array that lives on disk and is read in lazily: indexing it pages in
  only the touched regions instead of loading the whole file into RAM. The
  `DataLoader` needs scattered random windows, not a front-to-back scan, so a
  memmap lets it jump to any position on demand — and the corpus can be larger than
  the machine's RAM.

## Putting it together: the overnight run and sampling

With the tokenizer, the `DataLoader`, the GPT, and the loop all in hand, the
non-graded launcher `train.py` assembles stage 3 into the real run: it opens the
`train_tokens.bin` memmap, wraps it in your `DataLoader`, builds a `GPT` from a
config, and calls `train_loop` — streaming live metrics and checkpointing every
1000 steps so a relaunch can resume from the latest checkpoint via `train_loop`'s
`resume_from`. On an Apple-Silicon laptop the loss falls over a few hours; you
watch it drop.

Then stage 4, the payoff: load a checkpoint, feed the model a short prompt
("Once upon a time"), and call the `sample` function from L22 — it asks the model
for the next-token distribution, draws one token, appends it, and repeats. Out
comes a fresh little story in the model's own words. The generation-quality rubric
(`rubric_template.md`) walks you through scoring those samples honestly, and the
experiments make you predict the numbers before you measure them.

## Exercise

Implement in `my_work/tinystories-run/tinystories_run.py` (run
`uv run grade start L26-tinystories-run` to get the scaffold). The module has two halves: the
BPE warm-up (stdlib only, no torch) and the `DataLoader` (uses torch).

- `train_bpe(text, vocab_size)` — learn the merge rules. Parameters: `text` is the
  training string; `vocab_size` is the target vocabulary size, an int that must be
  `>= 256` (the 256 byte base). Train over the UTF-8 *bytes* of `text`; learn
  `vocab_size - 256` merges (fewer if no pair ever repeats). Returns an
  insertion-ordered `merges` dict mapping each merged byte pair `(a, b)` to its new
  id, where **insertion order is rank order** (the first merge learned gets id 256,
  the next 257, and so on).
- `encode(text, merges)` — turn a string into token ids. Parameters: `text` is the
  string to encode; `merges` is the dict from `train_bpe`. From the raw UTF-8
  bytes, repeatedly apply the present pair with the **lowest** merge rank
  (earliest-learned wins) until no learned pair remains. Returns a `list[int]` of
  ids. This re-applies the learned rules in rank order; it does **not** re-run
  training.
- `decode(ids, merges)` — turn ids back into the string. Parameters: `ids` is the
  `list[int]` from `encode`; `merges` is the same dict. Rebuild the id→bytes vocab
  (0..255 are single bytes; each merged id concatenates the bytes of its two
  parts), join the bytes for every id, and UTF-8-decode. Returns the original
  `str` — byte-level BPE is lossless, so `decode(encode(t)) == t` exactly.
- `DataLoader(tokens, block_size, batch_size)` — the batch source for training.
  Constructor parameters: `tokens` is a 1-D integer array (a numpy array or
  `np.memmap` of a `*_tokens.bin` file); `block_size` is the sequence length per
  window (an int); `batch_size` is how many windows per batch (an int). Store them
  and precompute `high = len(tokens) - block_size - 1` (the last in-range start),
  which you pass as `torch.randint`'s exclusive upper bound.
- `DataLoader.get_batch()` — produce one batch. Takes no arguments. Draw
  `batch_size` random start positions with `torch.randint(0, high, (batch_size,))`
  over the **global** torch RNG (no `generator=` argument), then for each start `i`
  slice `x = tokens[i : i + block_size]` and `y = tokens[i + 1 : i + 1 +
  block_size]`. Returns a tuple `(x, y)` of two `(batch_size, block_size)`
  `LongTensor`s, where `y` is `x` shifted one position forward (the next-token
  targets).

Then grade it: the site's Run Grader button or `uv run grade L26-tinystories-run`.

## How the grader checks you

The grader runs two suites, both on tiny synthetic data — CPU-only, deterministic,
seconds-fast, no real dataset and no overnight run.

The **warm-up suite** exercises the BPE half:

- **Round-trip** trains a BPE on a fixed repetitive corpus, then checks that
  `decode(encode(t))` returns `t` exactly for an ASCII case (`"hello"`) and an
  emoji case (`"world 🌍"`). The emoji case is the byte-level test: a
  character-based vocab cannot represent the multibyte emoji and fails here.
- **Merge-order trace** trains on the fixed corpus `"the the cat cat the"` with
  `vocab_size=260` and checks the learned merges equal, in order, `(104,101)→256`
  (`'he'`), `(116,256)→257` (`'the'`), `(32,99)→258` (`' c'`), `(32,257)→259`
  (`' the'`). Insertion order must equal rank order, and `encode` must apply merges
  in that rank order — not by encode-time frequency.
- **Compression** encodes the training corpus with a `vocab_size=400` BPE and
  checks the token-to-byte ratio falls below `0.70`; if `encode` barely shrinks the
  byte stream, it is not applying the learned merges.

The **pipeline suite** exercises the `DataLoader` and the inherited resume
guarantee:

- **Batch shapes** builds a `DataLoader` over an `arange` memmap (consecutive ids)
  and checks `get_batch()` returns `x` and `y` of shape `(batch_size, block_size)`
  with `y == x + 1` element by element — the next-token shift, made directly
  checkable by the consecutive ids.
- **Throughput floor** runs 20 `train_loop` steps on a 2-layer GPT over a tiny
  memmap and checks they clear a low steps-per-second sanity floor (0.5 steps/sec).
  This is a wiring check, not a performance gate — failing it usually means the
  `DataLoader` rebuilds heavy state every batch.
- **Resume contract** runs 10 steps uninterrupted to get a baseline, then runs 5
  steps with a checkpoint at step 5, reloads it via `resume_from`, runs 5 more, and
  checks the step-10 loss matches the baseline to `1e-6`. It also checks the
  checkpoint is Schema-v1 (`schema_version == 1`) and that `tokens_seen` equals
  `5 * batch_size`. This is where the RNG discipline is enforced (see the trap).

### The named trap: a private generator breaks exact resume

This grader is built to catch one mistake by name. If your `DataLoader` draws its
window starts from a **private** `torch.Generator()` (a fixed-seed generator you
create yourself) instead of the global torch RNG, the shapes and the `y = x + 1`
shift still come out right — so the batch-shapes check passes and nothing looks
wrong. But that private generator's state is never written into the checkpoint,
because `train_loop` only saves the *global* RNG state (`rng_cpu` / `rng_mps`).
After a reload the private generator produces different windows, so the resumed run
draws a different batch stream and its loss diverges from the uninterrupted run.
The resume-contract check fails and its message tells you the cause directly: the
`DataLoader` must *draw its window starts from the global torch RNG*. The fix is to
call `torch.randint(0, high, (batch_size,))` with no `generator=` argument.
