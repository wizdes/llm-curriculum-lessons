# L18 — Single-Head Self-Attention

The n-gram MLP (L17) hit a wall. To predict the next token it fed in the last `k`
tokens by **concatenation** — literally laying their embeddings end to end into
one long vector. (An **embedding** is the learned vector of numbers that
represents a token; a **token** is one unit of text, here a character.) That made
the first layer `k*d` wide, where `d` is the embedding length, so the parameter
count grew **linearly** with the context length `k`: doubling how far back the
model looks doubles the weights in that layer. Worse, what the model learned about
a token sitting in one slot could not be reused when the same token appeared in a
different slot — each position had its own private stretch of weights.

**Self-attention** breaks both limits at once. The idea, in one sentence: each
position builds its representation by **looking at the other positions and pulling
in a weighted mix of what they hold**, where the weights are *computed from the
content* of the tokens — not fixed, not tied to a slot — and the same small set of
weights is shared across every position. "Self" means the sequence attends to
*itself*: the queries, keys, and values (defined in a moment) all come from the
same input tokens, not from some separate source. This is the single most
important operation in the course, so we derive it **by hand** on a tiny example
before writing a line of the layer.

The worked example is deliberately small: **3 tokens**, head dimension **`d` = 4**
(four numbers per token after projection). Three tokens is the fewest that still
shows the causal mask doing real work (the third row attends to all three
positions; the first attends to only one), and four dimensions is wide enough to
be a real dot product yet narrow enough to print. Every number in the derivation
is generated by `derivation/gen_derivation.py`, shown across six panels in
`derivation/panels.md`, and frozen in `derivation/reference_values.json`. The
grader checks your `attention` against exactly those numbers.

We open with a 30-second warm-up that re-derives the L10 `adam_bias_correction`
cold — recalling an old result from memory, with no notes. (Adam, from L10, is the
optimizer that keeps running averages of the gradient and its square; the
bias-correction step divides out a startup bias those averages have early on.)
Pulling it back from memory now, right before new material, is **spaced
retrieval** — practising recall keeps the optimizer muscle warm.

## What query, key, and value mean

Before any formula, here is the cast of characters. Attention starts from a stack
of token vectors and turns each one into three different vectors by multiplying it
by three different learned weight matrices. The three roles:

- **Query (`Q`)** — *"what am I looking for?"* The query is the vector that the
  current token uses to search the others. Reading position `i`'s query asks: what
  kind of information does token `i` want to gather?
- **Key (`K`)** — *"what do I offer?"* Each token also advertises itself with a
  key. A query matches a key well when the thing the query is looking for is the
  thing that key advertises.
- **Value (`V`)** — *"what will I pass on if attended to?"* Once a token has been
  selected (its key matched some query), the value is the actual content that gets
  mixed into the output. Keys decide *how much*; values decide *what*.

Each of `Q`, `K`, `V` is a stack of `T` rows (one per token) with `d_k` numbers in
each row, written as shape `(T, d_k)`. Here `T` is the sequence length (`T = 3`)
and `d_k` is the per-token width (`d_k = 4`); read `d_k` aloud as "d-sub-k," the
key/query dimension. In the worked example all three share the same width, so `V`
is also `(T, d_k)`; in general the value width `d_v` may differ, and we write `V`
as `(T, d_v)`. The query and key *must* share a width because we dot them
together; the value is free to be a different width because it is only ever summed,
never dotted against `Q`.

## The four-step anatomy

Scaled dot-product attention turns those three stacks into one output stack in
four steps. In words first, then the procedure:

1. Score every query against every key — a dot product per pair — to get a `T x T`
   grid of match strengths.
2. Scale the grid down by a fixed factor (the reason has its own section).
3. Mask out the future: in a causal model token `i` may not peek at tokens after
   it, so those scores are forced to `-inf`.
4. Softmax each row into weights that sum to 1, then take the weighted sum of the
   value rows.

As an explicit procedure, with the shape after each line:

```text
1. scores  = Q @ K.T                  # (T, T): scores[i,j] = query i · key j
2. scaled  = scores / sqrt(d_k)       # (T, T): same grid, shrunk by sqrt(d_k)
3. masked  = scaled with j > i -> -inf  # (T, T): causal — token i sees only j <= i
4. weights = softmax(masked, dim=-1)  # (T, T): row-wise; each row sums to 1
   out     = weights @ V              # (T, d_v): the weighted value mix
```

`@` is matrix multiply and `K.T` is `K` transposed (rows and columns swapped), so
`Q @ K.T` dots every query row with every key row. `dim=-1` means "along the last
axis," i.e. across the columns within each row, so each row becomes its own
probability distribution. Walking the four steps:

**Step 1 — scores.** Dotting every query row with every key row gives a `T x T`
grid: `scores[i, j]` is how strongly token `i`'s query matches token `j`'s key —
how much position `i` wants to attend to position `j`. The **dot product** of two
vectors (multiply matching entries, add them up) is large and positive when the
vectors point the same way, so a big score means "this key is offering what this
query wants." (See Panel 2 of the derivation.)

**Step 2 — scale.** Divide the whole grid by `sqrt(d_k)`, the square root of the
key dimension. This keeps the scores from growing too large as `d_k` grows; the
`## Scale factor` section below explains exactly why.

**Step 3 — mask.** In a **causal** language model — one that predicts each token
from only the tokens before it — position `i` may look at the present and the
past, never the future. We enforce that by setting the **strict upper triangle**
of the grid (every entry where the key index `j` is greater than the query index
`i`) to `-inf` *before* the softmax. "Strict upper triangle" means above the
diagonal but not including it, so position `i` can still attend to itself. Why this
must happen before softmax, and why `-inf` and not `0`, has its own section below.
(See Panel 4.)

**Step 4 — softmax and output.** The **softmax** turns each row of raw scores into
a set of non-negative weights that sum to 1 — a probability distribution over the
positions that row is allowed to see. It exponentiates each score and divides by
the row's total, so larger scores get larger shares. Because `exp(-inf) = 0`, the
masked future positions get exactly zero weight and drop out. (See Panel 5.) Then
each output row is the attention-weighted average of the value rows:

$$
\text{out}[i] = \sum_{j} \text{weights}[i, j] \cdot V[j]
$$

Read aloud: *output row `i` is the sum over `j` of (how much `i` attends to `j`)
times (value row `j`).* Token `i`'s new representation is a blend of the values of
the tokens it attended to. (See Panel 6.) Notice in the panel that row 0 comes out
*equal to* `V` row 0 — token 0 can attend to nothing but itself, so its only
weight is 1.0 on its own value.

A concrete trace makes this solid. In the 3-token derivation the masked, scaled
scores in row 1 are `[-0.493, -0.937, -inf]` — position 0, position 1, and the
masked future. Softmax over those gives weights `[0.6092, 0.3908, 0.0]`: row 1
attends 61% to the past token 0 and 39% to itself, and 0% to the masked future, as
it must. The output row 1 is then `0.6092 * V[0] + 0.3908 * V[1] + 0 * V[2]`, which
the panel computes as `[0.257, 1.014, -1.845, -0.265]`. Every number you implement
is checked against this exact sheet.

## Causal mask timing

The single most common attention bug is masking at the **wrong time** or with the
**wrong value**. The rule is: set the strict upper triangle to `-inf` **before**
the softmax. Two tempting variants are both wrong:

- **Masking too late** — softmaxing the full, unmasked scores so the future is
  still in the row. The trouble is that softmax *normalizes the whole row*: it
  divides by the sum of the exponentials of every entry, including the future ones,
  so each future position gets a real, non-zero weight. The grader's trap takes the
  bluntest form of this bug — it never masks at all — and the symptom is what makes
  it sneaky: every row still sums to 1, so it *looks* like a valid distribution and
  nothing crashes, yet the strict upper triangle is non-zero, which means weight is
  sitting on future positions and their values flow straight into the output.
  Information from the future has quietly contaminated the present, and a causal
  model that can see its own answer will cheat in training and fail at generation.
  Trying to patch it by zeroing the future *after* softmax does not save you: the
  kept weights were already divided by a denominator that counted the future, so
  they are wrong and no longer sum to 1. The only correct fix is to remove the
  future *before* softmax, so it never enters the denominator.
- **Masking with `0` instead of `-inf`.** Setting a future score to `0.0` does
  *not* exclude it. After softmax, `exp(0) = 1`, so that position keeps a full unit
  of unnormalized weight and competes with every real position for probability.
  Only `-inf` works, because `exp(-inf) = 0` drives the weight to exactly zero and
  truly removes the position.

The grader pins both. For the *timing* check it sets `V` to the identity matrix —
the matrix with 1s on the diagonal and 0s elsewhere — because then
`out = weights @ I = weights`, so the output rows *are* the attention-weight
matrix. It can then read the strict upper triangle of the output directly and check
it is zero (no future leak) while each row still sums to 1 (a valid distribution).
For the *`-inf` vs `0`* check it instead picks a small `Q`, `K`, `V` where the only
position that would survive the wrong `0`-mask carries an obviously wrong value, so
the masked output row gives the bug away.

## Scale factor

Why divide by `sqrt(d_k)`? The dot product of two `d_k`-dimensional vectors is a
sum of `d_k` independent product terms, so its **variance** — the typical spread of
its values around the mean — grows in proportion to `d_k`. (Variance adds when you
sum independent terms; `d_k` terms means roughly `d_k` times the spread of one.)
So as the head dimension `d_k` grows, the raw scores grow large in magnitude — the
typical size of a score scales with `sqrt(d_k)`, so going from `d_k = 4`
(`sqrt(d_k) = 2`) to `d_k = 16` (`sqrt(d_k) = 4`) roughly **doubles** the typical
score magnitude.

Feed large scores into softmax and it **saturates**: the largest exponential
dominates all the others, and the distribution collapses to a near **one-hot**
vector — almost all the weight piled on a single position, near-zero everywhere
else. A near one-hot attention can only *copy* one position; it has lost the
ability to *blend* across several, which is the whole point of attention. It also
hurts learning: where softmax is saturated its slope is almost flat, so the
gradients that train `Q` and `K` nearly vanish.

Dividing each score by `sqrt(d_k)` rescales the magnitude back toward unit variance
regardless of `d_k`. We use the *square root* of `d_k`, not `d_k` itself, because
variance scales with `d_k` while the typical *magnitude* (the standard deviation,
the square root of the variance) scales with `sqrt(d_k)` — dividing by `sqrt(d_k)`
is exactly what cancels that growth. With the divisor in place the softmax stays
soft and the gradients stay healthy as `d_k` changes. The grader checks that at
`d_k = 16` your attention weights stay well below a near one-hot (max weight under
0.95); drop the divisor and the softmax saturates and the check fires.

## From the function to the layer

`attention(Q, K, V, causal=True, return_weights=False)` is the operation in the
raw — you hand it the three stacks directly. In a real network the `Q`, `K`, `V`
are not given; they are **produced** from the token vectors by three *separate*
learned projections. `SelfAttentionHead` wraps the same four steps as a learnable
layer that owns those projections:

```python
class SelfAttentionHead(nn.Module):
    def __init__(self, n_embd, head_size, max_T=256):
        super().__init__()
        self.head_size = head_size
        # three SEPARATE bias-free linear maps: token width -> head width
        self.query = nn.Linear(n_embd, head_size, bias=False)
        self.key   = nn.Linear(n_embd, head_size, bias=False)
        self.value = nn.Linear(n_embd, head_size, bias=False)
        self.register_buffer("tril", torch.tril(torch.ones(max_T, max_T)))  # causal mask

    def forward(self, x):            # x is (B, T, n_embd): batch, time, embed width
        B, T, _ = x.shape
        q, k, v = self.query(x), self.key(x), self.value(x)     # each (B, T, head_size)
        scores  = q @ k.transpose(-2, -1) / math.sqrt(self.head_size)  # (B, T, T)
        mask    = self.tril[:T, :T] == 0                        # True on strict upper triangle
        scores  = scores.masked_fill(mask, float("-inf"))       # future -> -inf, BEFORE softmax
        weights = F.softmax(scores, dim=-1)                     # row-wise
        return weights @ v                                      # (B, T, head_size)
```

A few terms in that block. `nn.Linear(n_embd, head_size, bias=False)` is PyTorch's
learned matrix multiply: it maps each token's `n_embd`-long vector to a
`head_size`-long one, and `bias=False` means there is no added offset vector — a
projection should rotate and rescale, not shift, and dropping the bias is the
standard convention here. `B` is the **batch** dimension: the layer processes `B`
independent sequences at once, each `T` tokens long, which is why the shapes carry
a leading `B`. `torch.tril` builds a **lower-triangular** matrix of 1s (1 on and
below the diagonal, 0 above); `tril[:T, :T] == 0` is therefore True exactly on the
strict upper triangle — the future — which is what `masked_fill` sets to `-inf`.

The `tril` mask is registered as a **buffer**, not a parameter. A **parameter** is
a tensor the optimizer trains; a **buffer** is a tensor the module carries along —
it moves to the right device with the module and is saved in the state dict — but
that gradient descent never updates. The causal mask is a fixed geometric fact, not
something to learn, so a buffer is exactly right. This single head is the atom of
the transformer; stacking many of them in parallel and concatenating their outputs
is **multi-head attention** — the next lesson.

- `attention` is checked against the by-hand derivation — the single source of

  `1e-5`. A **shape check** runs `(T, d_k) = (5, 8)` queries/keys and
  `(T, d_v) = (5, 6)` values and requires `(5, 6)` out; it also builds a
  `SelfAttentionHead(n_embd=10, head_size=7)`, feeds a `(2, 4, 10)` input, and
  requires `(2, 4, 7)` out. A **causal-property check** confirms that output row 0
  is unchanged whether you run the full sequence or just the first token — token 0
  can only attend to itself, so later tokens must not affect it.

- Three trap tests each pin one design decision (described next). Two of them use
  the `V = identity` probe so the output rows *are* the attention weights; the third
  uses a hand-picked `V` that makes the bug's wrong value obvious.

### The named traps

This grader is a test-the-teacher grader: it runs three deliberately buggy
versions of the module, each with exactly one wrong line, and your real code must
*fail differently* from each. Each trap names one of the three correctness points
the lesson turns on:

- **`mask_after_softmax`** — softmaxes the full scores and never masks (or masks
  after). With `V = identity` the output rows are the weights, so the strict upper
  triangle comes out non-zero: future tokens leak. The grader's message is *"rows
  still sum to 1 but future tokens leak — check *when* you mask, not just *what*."*
- **`mask_with_zero`** — sets future scores to `0.0` instead of `-inf`. Because
  `exp(0) = 1`, the future keeps a full unit of weight; in the grader's `T = 3`
  case it dominates row 1 and drags in the wrong value. The message is *"Setting
  scores to 0 doesn't exclude positions — exp(0)=1. Use float('-inf')."*
- **`missing_scale_factor`** — drops the `/ sqrt(d_k)` divisor. At `d_k = 16` with
  large-magnitude `Q`, `K` the softmax saturates and the max weight climbs toward
  1.0 (near one-hot). The message is *"Attention weights are near one-hot — you're
  missing the scaling factor. Divide scores by math.sqrt(d_k) before softmax."*

If your `attention` is correct on all three counts — masks the strict upper
triangle to `-inf` before softmax, and scales by `sqrt(d_k)` — it passes all three.
